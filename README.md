# Data Pipeline with Apache Airflow (Sparkify Subproject)

This repository contains the project implementation for the **Data Pipelines with Apache Airflow** course from the Udacity **[Data Engineer Nanodegree](https://www.udacity.com/course/data-engineer-nanodegree--nd027)**. 

## Project Scenario
A startup called Sparkify wants to automate and monitor their data warehousing ETL on AWS using Apache Airflow. Their original data resides in AWS S3 as user activity logs and song metadata in JSON format.

While the underlying database schema and SQL transformation commands are defined, the main objective of this project is to build custom, reusable Airflow operators to automate the ETL lifecycle, handle staging execution from S3 to Redshift, populate a star schema warehouse, and monitor data health using the Airflow UI.

---

## Business Process & Data Requirements
The analytics team needs an automated workflow to process song-play behavior to help them understand what music their users are listening to. To support this, the pipeline builds an AWS Redshift Data Warehouse utilizing a Star Schema optimized for high-performance analytical aggregation and reporting.

### Engineering Tasks
* Configure and launch an AWS Redshift cluster.
* Set up AWS and Redshift connection hooks natively within Airflow.
* Orchestrate and monitor the end-to-end pipeline using an Airflow DAG.

## Pipeline Implementation Details

1. **Idempotent Table Creation:** The DAG triggers a `CreateTableOperator` for each target structure. Tables are only initialized if they do not already exist (`CREATE TABLE IF NOT EXISTS`).
2. **Execution Synchronization:** A dedicated sync task enforces a strict boundary ensuring all table definitions are ready in Redshift before any S3 data staging attempts begin.
3. **Backfill & Scheduling:** `catchup = True` and a `@hourly` `schedule_interval` are configured to backfill historical transactional events across the specific November 2018 data block.
4. **Flexible Staging Modes:** Staging operators default to a clean truncate-load pattern to wipe temporary records, with an optional boolean flag to run in append mode.
5. **Fact Table Append-Only Logic:** The `LoadFactOperator` defaults to append-only mode (`truncate=False`) to naturally accommodate streaming data growth, with an explicit configuration parameter to truncate if required.
6. **Dimension Table Refresh:** The `LoadDimensionOperator` defaults to truncating target destination data on every operational sequence to maintain a clean dimension state.
7. **Dynamic Data Quality Checks:** The `DataQualityOperator` processes a structural metrics dictionary dynamically looping through tables to validate row populations and verify that required primary keys do not contain `NULL` violations.

## Code Usage & Deployment

To run this pipeline in your own local or workspace Airflow environment:

1. Clone this repository into your Airflow home environment.
2. Place the main orchestration workflow (`final_project.py`) into your local `dags/` directory.
3. Move the plugin files (`stage_redshift.py`, `load_fact.py`, etc.) into your local `plugins/operators/` folder.
4. Move the query module (`sql_queries.py`) into your local `plugins/helpers/` folder.
5. Ensure your directory initializers (`__init__.py`) are present in both the `operators/` and `helpers/` folders to establish package scope.
6. Launch your Airflow Scheduler and Webserver.
7. Open the Airflow UI, navigate to Admin $\rightarrow$ Connections, and create your credentials:
   * **`aws_credentials`**: Amazon Web Services hook containing your access key and secret token.
   * **`redshift`**: Postgres hook configured with your Redshift cluster endpoint, database name, and master password.
8. Toggle the `udac_example_dag` switch to **On** to begin execution.
